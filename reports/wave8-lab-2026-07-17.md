# Wave 8 — creative delight pass, prototype-first — 2026-07-17

Branch `explore/wave8-lab` (throwaway — never merged as-is, per GG's instruction). Five
isolated demos at `/lab/1`–`/lab/5`, each a self-contained prototype GG reacts to
separately. **Nothing in this wave touches the production site** — `git status` on this
branch shows only new, additive files (`app/lab/`, `components/lab/`,
`content/lab-tfidf-samples.ts`); zero production files edited.

## The tension, and how it's resolved here

Waves 6/7 were deliberately benchmarked against calm, minimal sites (emilkowal.ski,
paco.me, rauno.me) — restraint was the fix for a diagnosed composition problem. This wave
pulls toward energy instead, per GG's read that the result feels dull. Resolution: every
demo below is an **addition** layered on the existing restrained foundation (same tokens,
same dark+indigo palette, same type scale) — not a reskin, not new colors, not new
typography. The "delight" is entirely in *behavior* (motion, real-time compute, physical
interaction feel), which is orthogonal to the *visual restraint* wave 6/7 established.
Nothing here re-introduces boxes, extra accent colors, or decorative chrome.

## Reference research note

GG named Vercel and Linear's product pages as the reveal-pattern reference. Static
DOM inspection post-hydration on both (`vercel.com/products/observability`, `linear.app`)
turned up no reusable stagger-timing values — production React apps don't leave
`animation-delay`/`transition-delay` inspectable in the rendered DOM once React's
committed the final state, and by the time a scripted inspection runs, entrance
animations have typically already completed. Lab 2's 40-60ms-per-item cascade and
expo-out easing curve are therefore a **qualitative visual match** to how those sites
feel, not an extracted constant — disclosed here rather than presented as measured.

## The five prototypes

### Lab 1 — Embedding-space visualization as a loading state ⭐ (GG's "highest-leverage")

Reuses the *exact* production asset (`/heat-toy-vocab.json`, 410 words, 72-dim
sentence-transformer embeddings, PCA-reduced retaining 72.3% variance — wave 3) and the
same cosine-similarity math as the shipped heat toy. The guess and the secret word are
plotted using the real 1st and 2nd principal components (dims 0/1 of the actual vector,
not a fabricated projection) — axis bounds computed from the full real 410-word corpus.
Submitting a guess triggers a deliberate ~450ms reveal (guess point animates from its
previous position to its real new one via a CSS transform transition) — computation is
already complete in <5ms; the reveal is disclosed as staged pacing of a real result, not
a disguised wait.

Real numbers, verified live: guessing "fire" → cosine similarity **-0.083** (all 72 dims,
same value the production Cold/Warm/Hot label uses) plotted at real PC1=0.05, PC2=-0.02;
"ocean" → **-0.037** at PC1=-0.24, PC2=0.04. The similarity number and the 2D position
both come from the same real vectors — the plot is disclosed as a simplified 2-of-72-dim
view, never presented as the full similarity metric.

**Recording**: `reports/screenshots/wave8-lab/lab1-embedding-plot.gif` (6 frames: empty
plot → "fire" settled → "ocean" settled → two genuine mid-transition frames showing the
point interpolating between real coordinates → "fire" re-settled). The mid-transition
frames are real, not staged for the recording — captured by making the transition
temporarily 20-120s long (session-only, via a `<style>` override) so tool round-trip
latency (which varied 8-37s across calls in this environment) couldn't outrun it; see
"Recording methodology" below.

**Perf cost**: **2,369 bytes gzip marginal** (route-specific code only, beyond the
~185KB shared framework baseline every route pays — see methodology). One client fetch of
the already-shipped 41.7KB vocab JSON, only on this route. Main-thread cost: cosine sim
over 410 words is <5ms measured; the CSS transform transition is compositor-only (GPU),
no main-thread animation work.

### Lab 2 — Staggered stream-in reveal

The real skill-chip list (`content/about.ts`) and a real Experience bullet, revealed two
ways side by side: (A) whole-block at once — literally what production does today, since
wave 6 removed per-element reveal entirely; (B) per-element 50-60ms cascade.

**The wave-6 tension, addressed directly**: wave 6 deleted its predecessor's reveal effect
because it left the entire page blank without JavaScript (opacity:0 by default, only
`.animate()`-style disclosed in a class toggle that ran too late — reports/wave6-audit,
finding 8). This prototype uses a different technique: `Element.animate()` (Web
Animations API) with `fill: "backwards"`, called only from inside an `IntersectionObserver`
callback. The DOM's default state is **fully visible, always** — no CSS rule or
pre-hydration class ever sets opacity:0. No-JS, slow-hydration, and reduced-motion users
all see the plain content immediately; the entrance motion is a bonus, never a
precondition for visibility. This is a concrete technical answer to "how do we get motion
back without repeating the wave-6 bug," not just an assertion.

**Recording**: `reports/screenshots/wave8-lab/lab2-stagger-reveal.gif` — captured via the
same slow-motion technique (animate() durations ×15 for the recording only), clearly
showing row B's last chip ("FastAPI & GCP Cloud Run") still fading in while row A's whole
block already popped instantly.

**Perf cost**: **775 bytes gzip marginal**. Entrance animation runs once per element on
first scroll-into-view, GPU-compositable properties only (opacity + transform), no layout
thrash.

### Lab 3 — Modern momentum slider

Replaces wave 5's retired APG carousel-with-arrows pattern. Leans on **native** browser
scrolling (`overflow-x: auto` + `scroll-snap-type: x mandatory`) for trackpad/touch
momentum and rubber-banding — free, real OS-level physics, not a JS drag simulation
(which would be heavier and worse than what the platform already provides). Peek-of-next
on both sides via `scroll-snap-align: center`. Mouse users get pointer-drag-to-scroll
added (native scrolling has no click-drag by default). Progress is a thin animated fill
bar + "N / 10" fraction counter, not a dot row (dots don't scale cleanly past ~6 items;
this site has 10).

**Tested**: keyboard arrows (confirmed via driven browser interaction — lands exactly on
each card, center-snapped, progress bar and counter update correctly); simulated touch
swipe at 390px (scroll position + snap settling confirmed); mouse-drag path implemented
via pointer events (`pointerType === "mouse"` gate so it doesn't fight native touch
scrolling). **Bug found and fixed during verification**: initial implementation reused
`role="group"` on `<li>` elements — the exact same axe violation (`aria-allowed-role`)
wave 5's original carousel hit and fixed. Caught by this wave's own axe pass, fixed with
the same APG div-based pattern wave 5 used, re-verified at 0 violations.

**Recording**: `reports/screenshots/wave8-lab/lab3-momentum-slider.gif` (initial state →
after arrow-key navigation, showing peek-both-sides + progress bar advancing to 03/10).

**Perf cost**: **1,318 bytes gzip marginal**. Trackpad/touch momentum costs 0 JS (native
compositor scrolling); only the progress-bar/counter update runs JS, rAF-throttled on
scroll.

### Lab 4 — Live TF-IDF classify

A real TF-IDF classifier — tokenization, TF-IDF weighting, cosine similarity — computed
live in the browser, no server call, over 12 real, currently-public GitHub issue titles
(6 from `kubernetes/kubernetes`, 6 from `microsoft/vscode`, fetched via GitHub's public
search API 2026-07-17, sourced per-item with issue number + URL in
`content/lab-tfidf-samples.ts`). Mirrors TriageIQ's real first-stage technique on the
same two repos its published metric covers, **explicitly labeled as an illustrative
reproduction** — not the production model (no BGE+FAISS retrieval, no LightGBM, no Groq
synthesis, 12 sample docs vs. the real gold-labeled eval set) — so the delight doesn't
misrepresent what TriageIQ actually does.

**Verified live**: "kubernetes publishing bot is broken" → k8s 0.406 / vscode 0.000.
"Code Insider complains…typings files" (a real vscode sample) → k8s 0.044 / vscode 0.310.
Free-typed text "terminal crash on extension install" → k8s 0.000 / vscode 0.356 (correct
discrimination on novel input, not just the canned samples — the classifier is genuinely
live, not a lookup table).

**Recording**: `reports/screenshots/wave8-lab/lab4-tfidf-classify.gif` (k8s sample
classified → free-text guess classified, scores animating via the bar fill).

**Perf cost**: **2,194 bytes gzip marginal** (component + the 12-item corpus, inlined —
no network fetch). TF-IDF computation over 12 short titles: sub-millisecond, imperceptible
on any keystroke/click.

### Lab 5 — Scroll-linked real-data reveal

The actual wave-7 eval figures (Warmer's dumbbell, Style Maitri's bar, TriageIQ's paired
bars — same real sourced values, same `content/products.ts` figure fields) redrawn to
animate from 0 to their real value as each scrolls into view, instead of rendering
static. A self-contained duplicate for prototyping (production `eval-figure.tsx`
untouched); if advanced, the entrance behavior would move into the production component
directly rather than staying a fork.

**Recording**: `reports/screenshots/wave8-lab/lab5-scroll-draw.gif` — a genuine 3-frame
sequence (0% — tracks only, no bars — → ~48% visual progress — bars visibly half-drawn →
100% — settled at real values 93.8%, 82.5%, 90.4%). Captured via `Animation.pause()` +
manually scrubbed `currentTime` on the real `Element.animate()` calls (not simulated) —
this is a standard, deterministic Web Animations API technique for testing motion, not a
fabricated frame; see methodology below. Notable finding from this exercise: the
`cubic-bezier(0.16,1,0.3,1)` expo-out easing front-loads visual progress so heavily that
40% of *time* already looks ~94% *drawn* — worth knowing if this easing curve is reused
elsewhere.

**Perf cost**: **4,182 bytes gzip marginal** (largest of the five, still nowhere near
budget-relevant — three SVG figures' worth of `Element.animate()` calls plus
IntersectionObserver wiring). Entrance runs once per figure on scroll-into-view.

## Recording methodology (disclosed, not glossed over)

This harness has no native screen-recording tool — `take_screenshot` produces single
frames only. Two techniques were used to still capture genuine motion, not just static
before/after states, and every "recording" below is an assembled GIF from real captured
frames, not a continuous video:

1. **Slow-motion via temporary duration override** (Labs 1, 2): CSS transition durations
   or `Element.animate()` call durations were multiplied 15-300× for the capture session
   only (a `<style>` override or a `Element.prototype.animate` wrapper, removed
   afterward) so that this environment's variable tool round-trip latency (observed
   8-37 seconds between consecutive tool calls) couldn't finish the animation before the
   next screenshot. Production code is unaffected — the override lived only in the
   browser tab's live JS state during recording, never in a committed file.
2. **Deterministic scrub via the Web Animations API** (Lab 5): the `Animation` objects
   returned by `Element.animate()` were captured, `.pause()`-d immediately, and their
   `.currentTime` set directly to a chosen fraction — a standard technique for testing
   animation state precisely, independent of wall-clock timing entirely. This is more
   reliable than technique 1 and worth reusing for any future recording need in this repo.

Every frame in every GIF reflects a real, reachable application state — none are
mocked up separately from the running prototype.

  pattern), not extracted timing constants — disclosed above, not overstated.
- Lab 3's mouse-drag path was implemented and code-reviewed but not driven by an actual
